# REGENT Phase 0 cross-database contract

## Verified database matrix

Phase 0 is exercised against ephemeral databases only. No external or production database is contacted.

| Database | Verified image/version | Coverage |
| --- | --- | --- |
| H2 | 2.1.x (project runtime) | clean and legacy Flyway migration, import preflight, atomic replacement, generator alignment |
| PostgreSQL | `postgres:16-alpine` (16.14 in the recorded run) | legacy Flyway migration and full V6 export/rehearsal/import contract |
| MySQL | `mysql:8.0` (8.0.42 in the recorded run) | legacy Flyway migration and full V6 export/rehearsal/import contract |

MariaDB is recognized as part of the MySQL dialect family and uses the MySQL migration tree. The current real-container gate is MySQL 8.0; this change does not claim a separately verified MariaDB version.

## Flyway selection and SQL differences

`DatabaseFlywayConfiguration` reads the JDBC product name and selects exactly one migration location:

- `db/migration/h2`
- `db/migration/postgresql`
- `db/migration/mysql`

The migrations keep one logical version (`V1__phase0_security_identity.sql`) while using vendor syntax where required:

- H2 and PostgreSQL use `GENERATED BY DEFAULT AS IDENTITY`; MySQL uses `AUTO_INCREMENT` and InnoDB.
- PostgreSQL uses `BTRIM` and native conditional column/index DDL.
- MySQL 8 does not support the required `ADD COLUMN IF NOT EXISTS` form, so legacy upgrades use `information_schema` checks plus prepared DDL. Index creation is guarded the same way.
- Boolean expressions and account/profile backfill semantics are identical across all three trees.

## Import, identifiers, and generators

The save importer no longer clones an H2 schema with `SCRIPT NODATA`. It discovers the active catalog/schema, physical identifier case, quote character, tables, columns, and sequences through JDBC metadata. A complete import is rehearsed on a dedicated connection with autocommit disabled and is always rolled back. The live replacement remains one Spring transaction and preserves `users`, user-owned `person_profile` rows, and Flyway history.

Generator alignment is dialect-specific:

- H2 identity columns use `ALTER COLUMN ... RESTART WITH`; named generators use `ALTER SEQUENCE`.
- PostgreSQL `serial`/`bigserial` and SQL identity columns are discovered with `pg_get_serial_sequence` and aligned with `setval`; named generators use `ALTER SEQUENCE`.
- MySQL identity columns use `AUTO_INCREMENT`; Hibernate's named-sequence emulation tables (`cti_seq`, `scorer_seq`, `player_skills_seq`, `tphr_seq`) update their `next_val` row.

The calculated floor is the greater of the pre-import and imported row maxima. Generator operations happen before replacement DML because MySQL/H2 DDL can implicitly commit and PostgreSQL sequence changes are not transactionally rolled back. If later replacement DML fails, world/account rows roll back atomically; only generator counters may remain safely advanced.

## Boundaries

- `regent.enabled` remains `false` by default.
- This work does not add or change REGENT Phase 1 domain behavior.
- Compatibility means the tested Phase 0 Flyway and save/import contract above; it is not a production rollout or an assertion about untested database versions/configurations.
